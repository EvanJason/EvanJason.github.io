面试场景题

1. Git 分支管理场景​

1. dev 开发未完成,需紧急修复 test 分支 bug,Git 要怎么操作?​

场景描述: 你正在 dev 分支上进行新功能开发,代码尚未完成且未提交。此时测试团队在 test 分支发现了一个紧急 bug,需要立即修复并部署。

解决方案:

方案一:使用 Git Stash(推荐)​

# 1. 暂存当前 dev 分支的未提交工作
git stash save "WIP: feature development in progress"

# 2. 切换到 test 分支
git checkout test

# 3. 拉取最新代码
git pull origin test

# 4. 创建修复分支
git checkout -b hotfix/bug-fix

# 5. 修复 bug 并提交
git add .
git commit -m "fix: 紧急修复 xxx bug"

# 6. 推送到远程并合并到 test
git push origin hotfix/bug-fix
git checkout test
git merge hotfix/bug-fix
git push origin test

# 7. 切回 dev 分支继续开发
git checkout dev
git stash pop

优点:

  • 简单直接,保留工作现场
  • 可以随时恢复未提交的代码
  • 适合短期切换场景

注意事项:

  • stash pop 可能会产生冲突,需要手动解决
  • 建议给 stash 添加描述信息,方便后续查找

方案二:提交临时 commit​

# 1. 提交当前未完成的工作(使用 WIP 标记)
git add .
git commit -m "WIP: save work in progress before hotfix"

# 2. 切换到 test 分支修复 bug
git checkout test
git pull origin test
git checkout -b hotfix/bug-fix
# ... 修复 bug ...
git add .
git commit -m "fix: 紧急修复 xxx bug"
git push origin hotfix/bug-fix

# 3. 合并到 test 分支
git checkout test
git merge hotfix/bug-fix
git push origin test

# 4. 切回 dev 分支,撤销临时 commit
git checkout dev
git reset --soft HEAD~1  # 保留更改,撤销 commit
git reset HEAD .          # 取消暂存
# 或者使用 git reset --mixed HEAD~1(默认)

优点:

  • 更安全的保存方式
  • 可以通过 commit 历史追踪
  • 避免 stash 丢失风险

缺点:

  • 会污染 commit 历史
  • 需要额外清理临时 commit

方案三:创建新分支保存进度​

# 1. 创建临时分支保存当前进度
git checkout -b feature/wip-save

git add .
git commit -m "WIP: save progress before hotfix"

# 2. 切换回 dev 分支并重置
git checkout dev
git reset --hard HEAD

# 3. 去 test 分支修复 bug
git checkout test
git pull origin test
git checkout -b hotfix/bug-fix
# ... 修复 bug ...
git add .
git commit -m "fix: 紧急修复 xxx bug"
git push origin hotfix/bug-fix
git checkout test
git merge hotfix/bug-fix
git push origin test

# 4. 修复完成后,回到临时分支继续开发
git checkout feature/wip-save
# 继续开发...
# 开发完成后合并回 dev
git checkout dev
git merge feature/wip-save
git branch -d feature/wip-save

优点:

  • 最安全的方式
  • 完整的版本控制
  • 便于团队协作和代码审查

缺点:

  • 操作步骤较多
  • 会产生额外的分支

最佳实践建议:

  1. 优先使用 Stash:短期切换(几小时内)用 stash
  2. 重要工作用分支:长期切换(超过一天)用临时分支
  3. 避免频繁 stash:stash 过多容易混乱,定期清理
  4. 添加详细描述:git stash save "描述" 或 git stash push -m "描述"
  5. 查看 stash 列表:git stash list
  6. 应用特定 stash:git stash apply stash@{0}

2. 把 test 修复的 bug 合入 dev 分支有哪几个方法?​

场景描述: 在 test 分支上修复了紧急 bug 并已上线,现在需要将这些修复合并到 dev 分支,确保开发环境也包含这些修复。

解决方案:

方法一:直接 Merge(最简单)​

# 切换到 dev 分支
git checkout dev

# 拉取最新代码
git pull origin dev

# 合并 test 分支
git merge test

# 解决可能的冲突
git add .
git commit  # 如果有冲突

# 推送
git push origin dev

优点:

  • 操作简单直观
  • 保留完整的合并历史
  • 可以看到哪些 commit 来自 test

缺点:

  • 如果 test 分支有很多其他 commit,会一起合并过来
  • 可能引入不需要的代码
  • 产生合并 commit,历史线不够清晰

适用场景:

  • test 分支只有 bug 修复相关的 commit
  • 不介意合并 commit
  • 两个分支差异不大

方法二:Cherry-pick(精确选择)​

# 1. 查看 test 分支的 commit 历史
git log test --oneline
# 找到 bug 修复的 commit hash,例如: abc1234

# 2. 切换到 dev 分支
git checkout dev
git pull origin dev

# 3. Cherry-pick 特定的 commit
git cherry-pick abc1234

# 4. 如果有多个 commit,可以连续 cherry-pick
git cherry-pick abc1234 def5678 ghi9012

# 5. 或者指定范围(注意:不包含起始 commit)
git cherry-pick abc1234..xyz7890

# 6. 解决可能的冲突
git add .
git cherry-pick --continue  # 继续
# 或
git cherry-pick --abort     # 放弃

# 7. 推送
git push origin dev

优点:

  • 精确选择需要的 commit
  • 不会引入无关代码
  • 保持 dev 分支历史干净

缺点:

  • 需要手动查找 commit hash
  • 多个 commit 时操作繁琐
  • 可能产生冲突,需要逐个解决
  • 相同的修改在不同分支有不同的 commit hash

适用场景:

  • test 分支有大量无关 commit
  • 只需要部分修复
  • 希望保持历史简洁

高级技巧:

# 编辑 commit 信息
git cherry-pick -e abc1234

# 不自动提交,只应用更改
git cherry-pick -n abc1234

# 继续处理冲突后的 cherry-pick
git cherry-pick --continue

# 跳过当前有问题的 commit
git cherry-pick --skip

# 完全放弃
git cherry-pick --abort

方法三:Rebase + Merge(保持线性历史)​

# 1. 在 test 分支上 rebase dev
git checkout test
git pull origin test
git rebase dev

# 2. 解决可能的冲突
git add .
git rebase --continue

# 3. 切换到 dev 分支
git checkout dev
git pull origin dev

# 4. 快进合并(fast-forward)
git merge test

# 5. 推送(可能需要强制推送,谨慎使用)
git push origin dev

优点:

  • 保持线性历史,没有合并 commit
  • 历史清晰易读
  • 适合小型团队或个人项目

缺点:

  • 改变了 commit 历史
  • 如果 test 分支已推送到远程,会影响其他人
  • 需要团队成员理解 rebase 的概念
  • 公共分支不建议使用 rebase

适用场景:

  • 个人项目或小团队
  • test 分支未被其他人使用
  • 追求干净的提交历史

方法四:通过 Hotfix 分支(规范化流程)​

# 1. 从 dev 创建 hotfix 分支
git checkout dev
git pull origin dev
git checkout -b hotfix/urgent-bug-fix

# 2. 从 test 分支 cherry-pick bug 修复
git cherry-pick <commit-hash-from-test>

# 3. 或者手动应用修复(复制代码)
# ... 手动修复 bug ...

# 4. 提交并测试
git add .
git commit -m "fix: 紧急修复 xxx bug"

# 5. 合并到 test(用于验证)
git checkout test
git merge hotfix/urgent-bug-fix
git push origin test

# 6. 合并到 dev
git checkout dev
git merge hotfix/urgent-bug-fix
git push origin dev

# 7. 删除 hotfix 分支
git branch -d hotfix/urgent-bug-fix
git push origin --delete hotfix/urgent-bug-fix

优点:

  • 流程规范,可追溯
  • 便于代码审查
  • 可以在 hotfix 分支进行测试
  • 符合 Git Flow 工作流

缺点:

  • 步骤较多
  • 需要创建额外分支
  • 适合有严格流程的团队

适用场景:

  • 大型团队项目
  • 有严格的代码审查流程
  • 遵循 Git Flow 或 GitHub Flow

方法五:使用 Git Patch(特殊场景)​

# 1. 在 test 分支生成 patch 文件
git checkout test
git format-patch -1 <commit-hash>  # 生成单个 commit 的 patch
# 或
git diff commit1 commit2 > bug-fix.patch  # 生成差异补丁

# 2. 切换到 dev 分支
git checkout dev
git pull origin dev

# 3. 应用 patch
git apply bug-fix.patch

# 4. 检查是否有问题
git status

# 5. 提交
git add .
git commit -m "fix: 紧急修复 xxx bug (from test)"

# 6. 推送
git push origin dev

优点:

  • 不需要访问远程仓库
  • 可以离线操作
  • 适合跨仓库传递修复

缺点:

  • 操作复杂
  • 容易产生冲突
  • 不保留 commit 元数据(作者、时间等)

适用场景:

  • 跨仓库传递修复
  • 网络受限环境
  • 特殊的协作场景

方法对比总结:

方法复杂度安全性历史清晰度适用场景
直接 Merge⭐⭐⭐⭐⭐⭐分支差异小,快速合并
Cherry-pick⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐精确选择 commit
Rebase + Merge⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐个人项目,追求线性历史
Hotfix 分支⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐⭐团队规范流程
Git Patch⭐⭐⭐⭐⭐⭐⭐⭐⭐跨仓库/离线场景

最佳实践建议:

  1. 日常开发:优先使用 Cherry-pick,精确控制合并内容

  2. 团队协作:使用 Hotfix 分支流程,便于代码审查和追溯

  3. 紧急情况:直接 Merge 最快,但要注意审查变更

  4. 避免 Rebase 公共分支:除非团队所有人都理解并接受

  5. 合并前检查:

    # 预览将要合并的内容
    git log dev..test --oneline

    # 查看具体差异
    git diff dev...test

    # 试运行合并(不实际执行)
    git merge --no-commit --no-ff test
    git merge --abort  # 如果有问题就取消

  6. 解决冲突技巧:

    # 使用图形化工具
    git mergetool

    # 查看冲突文件
    git status

    # 接受我们的版本
    git checkout --ours <file>

    # 接受他们的版本
    git checkout --theirs <file>

  7. 合并后验证:

    # 运行测试
    npm test

    # 构建检查
    npm run build

    # 本地启动验证
    npm run dev

常见陷阱:

❌ 不要这样做:

# 错误:在公共分支上使用 force push
git push --force origin dev

# 错误:合并后不测试就直接推送
git merge test
git push origin dev  # 没有运行测试!

# 错误:忽略合并冲突
git merge test
# 看到冲突提示,却直接 commit

✅ 应该这样做:

# 正确:先拉取最新代码
git pull origin dev

# 正确:合并前预览
git log dev..test --oneline

# 正确:合并后运行测试
git merge test
npm test
npm run build
git push origin dev

# 正确:遇到问题及时沟通
git merge --abort  # 取消合并
git log test --oneline  # 重新检查
team_discuss()  # 与团队讨论

面试加分点:

  • 理解不同方法的优缺点和适用场景
  • 能够根据团队规模和项目特点选择合适的策略
  • 熟悉 Git 冲突解决的最佳实践
  • 了解 Git Flow、GitHub Flow 等工作流
  • 强调合并前的代码审查和测试验证
  • 提到使用 CI/CD 自动化验证合并结果
